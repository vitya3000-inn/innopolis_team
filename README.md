# NewsMap

Мобильное приложение (Expo / React Native) и Node.js backend: сбор политических новостей из NewsAPI, объединение статей в **события**, раскладка по **темам**, обогащение текстов через **OpenAI**, опционально — перевод на русский и история прогонов в **Supabase**.

---

## О проекте

### Назначение

Показать пользователю **не ленту из сотен заголовков**, а **ограниченный набор тем** (в MVP — три фиксированных блока: Иран, Украина, Израиль/Газа/Ливан) и внутри каждой темы — **события**, в которых уже **склеены** материалы нескольких СМИ, плюс **AI summary**, ключевые факты и **позиции источников** (карточки по каждому медиа).

### Стек

| Часть | Технологии |
|--------|------------|
| Клиент | Expo ~54, React Native, React Navigation |
| Сервер | Node.js, `http`, без Express |
| Данные | `backend/data/store.json` (текущая лента), опционально PostgreSQL через Supabase |
| Внешние API | NewsAPI, Google Translate (бесплатный endpoint), OpenAI Chat Completions |

### Архитектура в двух словах

1. **Backend** периодически и по запросу выполняет **refresh**: тянет статьи, строит события и темы, пишет **`store.json`** и при включённом флаге — **Supabase**.
2. **Приложение** ходит по HTTP на backend: **`GET /topics`**, **`GET /topics/:id/events`**, **`GET /events/:id`** (без даты — из `store.json`; с **`?date=`** — архив из Supabase, см. ниже).
3. **Секреты** (NewsAPI, OpenAI, Supabase service role) живут только на машине с backend; в клиент уходит только URL API (`EXPO_PUBLIC_API_URL` и т.д.).

## Пайплайн данных (как всё связано)

Цепочка реализована в **`backend/pipeline.js`** (`refreshFromSources`).

1. **Загрузка статей** — **`newsApi.js`**: запросы к NewsAPI по списку источников и доменов, общий поисковый запрос, дополнительные «доборные» запросы по темам, дедупликация по URL, кэш на диске (`newsapi-cache.json`). Параметры: окно **`NEWS_LOOKBACK_DAYS`**, лимит **`MAX_ARTICLES_PER_SOURCE`**, задержка между запросами.

2. **Перевод (опционально)** — при **`TARGET_LANGUAGE=ru`** заголовок, описание и фрагмент `content` проходят через **`translate.js`** (Google). Перед переводом текст очищается от служебных хвостов NewsAPI — **`newsTextCleanup.js`**.

3. **Подтягивание полного текста (опционально)** — при **`NEWS_ENRICH_ARTICLE_BODY=1`** для усечённых статей делается HTTP-загрузка страницы и грубое извлечение абзацев — **`articleBodyEnrich.js`** (ограничение числа запросов за прогон).

4. **Кластеризация статей → события**
   - Если заданы **`OPENAI_API_KEY`** и **`OPENAI_ARTICLE_GROUPING=1`**, используется **LLM-группировка** — **`articleLlmGroup.js`**: Union–Find по батчам OpenAI, опционально «целый бакет» темы, пост-склейка синглтонов и т.д.
   - Иначе — эвристическая кластеризация — **`cluster.js`** (`clusterArticlesIntoEvents` и родственные функции).

5. **Обогащение событий LLM** — **`llmOpenAI.js`** (`enrichEventsWithLLM`): краткий summary, keyFacts, analysis, релевантность к теме; опционально второй проход проверки секции. Кэш на диске (`llm-cache.json`). Лимит событий за прогон — **`OPENAI_MAX_EVENTS_PER_REFRESH`**.

6. **Сборка тем для UI** — **`cluster.js`** (`groupEventsIntoTopics`): при наличии ключа OpenAI на главный экран попадают **только три фикс-темы**; события с низкой **`llmTopicRelevance`** отфильтровываются; на тему действует лимит **`MAX_EVENTS_PER_TOPIC`**.

7. **Сохранение**
   - Весь результат ленты — **`store.js`** → **`backend/data/store.json`**.
   - В Supabase — только то, что **реально видно в приложении**: события из итоговых тем и статьи, которые в этих событиях есть в `opinions` — **`supabasePersist.js`**.

8. **Периодический refresh** — в **`server.js`** после старта сервера вызывается refresh и далее по интервалу **`refreshIntervalMs`** (в коде по умолчанию **24 часа**). Свайп «обновить» в приложении **повторно не запускает** пайплайн — только перечитывает JSON с сервера; полная пересборка — **`POST /admin/refresh`** (см. ниже).

## Мобильное приложение

- **Навигация:** при настроенном Supabase Auth — сначала экран входа/регистрации, затем `Feed` → `Topic` → `Event` (`App.tsx`).
- **Главный экран** — темы, фильтр по категории, календарь архива; у вошедшего пользователя в шапке — email и выход.
- **Экран темы** — список событий; **экран события** — AI summary, факты, карточки СМИ со ссылками на оригиналы.
- **API-клиент** — **`src/services/newsApi.ts`**: базовый URL из **`EXPO_PUBLIC_API_URL`** / Expo extra / LAN.

## HTTP API backend

| Метод | Путь | Назначение |
|--------|------|------------|
| GET | `/health` | Проверка живости |
| GET | `/topics` | Список тем; опционально **`?date=YYYY-MM-DD`** (архив UTC, Supabase) |
| GET | `/topics/:topicId/events` | События темы; тот же **`?date`** |
| GET | `/events/:eventId` | Одно событие; тот же **`?date`** |
| POST | `/admin/refresh` | Полный прогон пайплайна; заголовок **`Authorization: Bearer <Supabase access_token>`** и email из списка админов (см. ниже). **`?force=1`** — обойти свежесть кэша NewsAPI. Для отладки без JWT: **`ADMIN_REFRESH_OPEN=1`** в `.env` backend |
| GET | `/admin/visit-stats?from=YYYY-MM-DD&to=YYYY-MM-DD` | Число записей в **`app_visits`** за календарный диапазон **UTC** (границы включительно по дням). Тот же **Bearer** и проверка админа. Таблица создаётся скриптом **`backend/supabase/app_visits.sql`** |

## Переменные окружения (обзор)

Полный список и значения по умолчанию — в **`backend/config.js`**. Типичный **`.env`** в корне проекта:

- **`NEWS_API_KEY`** — обязателен для загрузки новостей.
- **`PORT`** — порт backend (по умолчанию 8787).
- **`TARGET_LANGUAGE`** — например `ru` для перевода.
- **`OPENAI_API_KEY`**, **`OPENAI_MODEL`**, **`OPENAI_ARTICLE_GROUPING`**, **`OPENAI_MAX_EVENTS_PER_REFRESH`** и др. — LLM и лимиты.
- **`SUPABASE_ENABLED=1`**, **`SUPABASE_URL`**, **`SUPABASE_SERVICE_ROLE_KEY`** — запись и чтение архива (service role только на сервере).
- **`EXPO_PUBLIC_API_URL`** (или extra в `app.config.js`) — URL backend для приложения.
- **`SUPABASE_URL`** + **`SUPABASE_ANON_KEY`** в `.env` достаточно для Auth: при старте Expo **`app.config.js`** копирует их в **`EXPO_PUBLIC_SUPABASE_*`**, иначе в JS-бандл не попадают только `SUPABASE_*` (Metro отдаёт клиенту лишь `EXPO_PUBLIC_`). При желании задайте **`EXPO_PUBLIC_SUPABASE_URL`** / **`EXPO_PUBLIC_SUPABASE_ANON_KEY`** вручную — они имеют приоритет. **Не используйте `SUPABASE_SERVICE_ROLE_KEY` в клиенте** — только anon.

### Регистрация и вход (email / пароль)

Если заданы **URL проекта** и **anon key** (любой из вариантов выше), при старте показывается экран **входа** (email + пароль, `signInWithPassword`) и **регистрации** (`signUp`), опционально **сброс пароля** по email. Сессия хранится в **`AsyncStorage`**.

В панели Supabase: **Authentication → Providers → Email** включён по умолчанию. Для разработки можно отключить **«Confirm email»**, чтобы сразу получать сессию после регистрации; иначе пользователь должен подтвердить почту по ссылке из письма.

Если **нет URL и anon key для клиента** (частая ситуация: в `.env` есть только `SUPABASE_URL` и `SUPABASE_SERVICE_ROLE_KEY` для backend, а **anon** не добавлен), приложение сразу открывает ленту без авторизации. Добавьте **`SUPABASE_ANON_KEY`** или **`EXPO_PUBLIC_SUPABASE_ANON_KEY`** и перезапустите Expo: **`npx expo start -c`**.

Код: **`src/lib/supabase.ts`**, **`src/contexts/AuthContext.tsx`**, **`src/screens/AuthScreen.tsx`**.

## Календарь и архив по дню

### Зачем это нужно

Позволяет открыть **не «свежую» ленту с диска сервера**, а **снимок ленты за выбранный календарный день** — тот, который был сохранён при очередном **ingestion** (refresh) в **Supabase**.

Важно: день считается по **UTC** на стороне backend при запросе к БД. В приложении дата выбирается в **нативном календаре** (iOS / Android); в веб-сборке используется ручной ввод `YYYY-MM-DD`.

### Что такое «день» для архива

- В БД при каждом успешном refresh создаётся строка в таблице **`ingestion_runs`** с полем **`started_at`** (момент старта прогона, ISO UTC).
- Для запроса **`?date=YYYY-MM-DD`** backend берёт интервал **`[00:00:00 UTC; следующий день 00:00:00 UTC)`** и выбирает **один** прогон — **последний по времени** среди всех refresh за этот UTC-день (если было несколько тестов или ручных обновлений).
- Затем подгружаются все **`events`**, привязанные к этому `run_id`, и из их `payload` заново собирается структура тем (как при live-режиме).

Это **не** «новости по дате публикации в источнике», а **«что было в приложении после прогона, который попал в этот UTC-день»**.

### Как подгрузить данные за конкретный день в приложении

1. Убедитесь, что backend доступен приложению (`EXPO_PUBLIC_API_URL` / порт, тот же Wi‑Fi для телефона).
2. В Supabase должны быть включены **`SUPABASE_ENABLED=1`**, **`SUPABASE_URL`**, **`SUPABASE_SERVICE_ROLE_KEY`** на сервере с backend (persist пишет историю).
3. На **главном экране** нажмите кнопку с иконкой календаря (**«Дата (UTC)»** / выбранная дата).
4. В модалке выберите нужный день в календаре и нажмите **«Готово»**.
5. Список тем перезагрузится с параметром **`?date=YYYY-MM-DD`** к API.
6. При переходе в тему и в карточку события выбранная дата **пробрасывается** — запросы идут с тем же `date`, чтобы показывать согласованный архивный снимок.

**Вернуться к «живой» ленте:** в той же модалке нажмите **«Актуальная лента»** (или аналог сброса) — запросы пойдут без `date`, данные читаются из **`backend/data/store.json`** после последнего refresh.

### Если за выбранный день в БД нет прогонов

Backend отдаёт **три пустые фикс-темы** (как заглушки) и в **`meta`** может быть **`historicalMissing: true`**. Событий в темах не будет — это ожидаемо, пока за этот UTC-день не было успешной записи в `ingestion_runs`.

### Тот же API вручную (curl)

```bash
# Актуальная лента (без даты)
curl -s "http://127.0.0.1:8787/topics"

# Архив за календарный день UTC
curl -s "http://127.0.0.1:8787/topics?date=2026-04-10"

# События темы и одно событие
curl -s "http://127.0.0.1:8787/topics/topic-fixed-iran-war/events?date=2026-04-10"
curl -s "http://127.0.0.1:8787/events/ev-xxxxx?date=2026-04-10"
```

Порт замените на свой **`PORT`** из `.env`, если не `8787`.

### Почему два разных дня могут «выглядеть» похоже

Источники и запросы к NewsAPI часто возвращают **одни и те же крупные истории** несколько дней подряд. Идентификаторы событий (`client_event_id`) и прогон (`ingestionRunId` в метаданных архива) при этом могут отличаться — сравнивайте при необходимости через ответ API или таблицу `ingestion_runs` в Supabase.

## Быстрый запуск

```bash
npm install
# в корне: .env с NEWS_API_KEY, при необходимости OPENAI_* и SUPABASE_*

npm run backend   # API на PORT (по умолчанию 8787)
npm start         # Expo
```

Принудительный refresh пайплайна (обновляет `store.json` и при включённом Supabase пишет новый прогон):

```bash
curl -X POST "http://127.0.0.1:8787/admin/refresh?force=1"
```

Схема таблиц Supabase: **`backend/supabase/schema.sql`**. Полный сброс данных ingestion (осторожно): **`backend/supabase/reset_all_data.sql`**.
